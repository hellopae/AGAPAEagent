# SOP-08 — QA Standards: เกณฑ์ตรวจของ Chris (และ Reese Fact-check) ฉบับเต็ม

> Chris ใช้ไฟล์นี้เป็น checklist หลัก | Claudy ใช้เช็คว่า verdict ของ Chris ครบถ้วนไหม
> หลักการ: **Opinions pass. Unverified facts don't.**

## ลำดับ gate ที่ถูกต้อง

```
ผลงาน → Reese [Fact-check] (ถ้ามี factual claims) → Chris QA → ship
```
Chris จะไม่รับงานที่มี factual claims แต่ไม่มีรายงาน fact-check แนบ — ตีกลับทันที

## A. เกณฑ์ Fact-check ของ Reese (สรุปจาก scaffold)

- ทำ claim inventory ทุกข้อ: สถิติ, วันที่, เหตุการณ์, คำกล่าวอ้างวิทยาศาสตร์, spec เทคนิค, ตัวเลขตลาด
- Verdict รายข้อ: ✅ VERIFIED (พร้อม source) / ⚠️ UNVERIFIED / ❌ INCORRECT (พร้อมข้อมูลที่ถูก) / 💬 OPINION
- Overall: PASS = ทุกข้อ ✅ หรือ 💬 | FAIL = มี ❌ หรือ ⚠️ ที่ยังไม่แก้
- ⚠️ แก้ได้ 2 ทาง: หา source มายืนยัน หรือเขียนใหม่ให้เป็นความเห็นชัดเจน

## B. Checklist ภาษาไทย (งานเขียนทุกชิ้น)

- [ ] สะกดถูกทุกคำ (ระวังคำพ้อง: ปฏิทิน/ปดิทิน, อนุญาต/อนุญาติ, สังเกต/สังเกตุ)
- [ ] ไม่มีสำนวนแปล ("มันเป็นสิ่งสำคัญที่จะ...", "หนึ่งในสิ่งที่...") — ต้องอ่านเหมือนคนไทยเขียน
- [ ] ระดับภาษาตรงบริบท: พิธีกรรม = สุภาพ-สูง / สถาบัน = ทางการ / listing-บทความ = อบอุ่นเป็นกันเอง
- [ ] เว้นวรรคถูกตามหลักราชบัณฑิตฯ ไม่มี double space
- [ ] คำทับศัพท์สะกดตามราชบัณฑิตฯ หรือคงอังกฤษไปเลย — ห้ามสะกดมั่ว
- [ ] ตัวเลขไทย/อารบิกใช้แบบเดียวกันทั้งชิ้น

## C. Checklist วัฒนธรรม/ศาสนา (ความเสี่ยงสูงสุดของธุรกิจ)

- [ ] คำศัพท์พุทธถูกต้อง (บาลี-สันสกฤต: กฐิน, สังฆทาน, อาสาฬหบูชา ฯลฯ)
- [ ] วันสำคัญตรงปฏิทินจริงของปีนั้น (วันพระ/วันสำคัญเลื่อนตามจันทรคติ — ต้องมี source ยืนยัน)
- [ ] Iconography เหมาะสม: ตำแหน่งพระพุทธรูป/สัญลักษณ์ศาสนาต้องอยู่ในบริบทเคารพ
  (ห้ามอยู่ต่ำกว่าข้อความ, ห้ามใช้เชิงตลก/ตกแต่งพร่ำเพรื่อ)
- [ ] ราชวงศ์/สถาบัน: ถ้าแตะต้อง ราชาศัพท์ต้องถูก 100% — ไม่มั่นใจ = escalate Kittanate
- [ ] สีมีความหมายทางวัฒนธรรม (ดำ/ขาว = งานอวมงคล) — เช็คว่าเหมาะกับโอกาสของชิ้นงาน

## D. Checklist Print Spec

- [ ] **ขนาดตรง dieline/asset ต้นทางจริง** — cross-check กับไฟล์ต้นทาง (เช่น `Box4`) ไม่ใช่แค่ตัวเลขที่ agent กรอกมา (บทเรียน VeeGee: ออกแบบผิดขนาด)
- [ ] **ใช้ asset ต้นทางที่มีจริงครบ** — ภาพถ่ายสินค้า/โลโก้ vector ที่มีในโฟลเดอร์ต้องถูกใช้จริง ห้ามแทนด้วยของวาดเอง (บทเรียน VeeGee: มีภาพจริงแต่ใช้ SVG แทน)
- [ ] ขนาดตรง spec ของ Vera (mm) | bleed 3mm ครบทั้ง 4 ด้าน
- [ ] Text/element สำคัญอยู่ใน safe margin (≥5mm จาก trim)
- [ ] CMYK เท่านั้น — เจอ RGB = blocker | 300dpi ขึ้นไป
- [ ] ฟอนต์ไทย render ถูก: สระ-วรรณยุกต์ไม่ลอย/ไม่ชนกัน | ฟอนต์ embed ได้หรือ outline แล้ว
- [ ] Export เป็น PDF/X (หรือ format ที่โรงพิมพ์/แพลตฟอร์มระบุ)
- [ ] Contrast พอสำหรับการพิมพ์จริง (สีอ่อนบนพื้นอ่อน = warning)

## E. Checklist Web

- [ ] ทุก flow หลักทำงานตาม Vera spec รวม loading/error/empty states
- [ ] Responsive: มือถือ (จอแคบสุด 360px) ถึง desktop
- [ ] ภาษาไทย render ถูก: ฟอนต์โหลดจริง, line-height ≥1.6, ไม่ตัดกลางคำ
- [ ] `npm run build` ผ่าน + ทดสอบบน URL จริงหลัง deploy (ไม่ใช่แค่ localhost)
- [ ] Console ไม่มี error แดงค้าง

## F. Checklist Listing (Etsy/Gumroad)

- [ ] Title มี keyword หลัก + อ่านรู้เรื่อง (ไม่ยัด keyword จนเละ)
- [ ] Description: ย่อหน้าแรกขายประโยชน์ต่อผู้ซื้อ, ระบุสิ่งที่ได้รับ (format, ขนาด, จำนวนไฟล์)
- [ ] Tags/keywords ครบตาม research ของ Reese
- [ ] ราคาอยู่ในกรอบ $5–25 และสอดคล้อง pricing research
- [ ] ระบุชัดว่าเป็น digital download — ไม่มีการจัดส่งของจริง

## รูปแบบ Verdict ของ Chris (บังคับ)

```
**Verdict:** ✅ PASS / ❌ FAIL
**Blockers:** (ต้องแก้ก่อน ship — ระบุตำแหน่ง + สิ่งที่ผิด + ควรเป็นอะไร)
**Warnings:** (ควรแก้)
**Notes:** (เล็กน้อย)
```
- Blocker ทุกข้อต้องชี้ตำแหน่งชัด — "โดยรวมยังไม่ดี" ถือว่า Chris ทำงานไม่ครบ ตีกลับให้ Chris ทำใหม่
- งานที่แก้ blocker แล้วต้องเข้า gate ใหม่**ตั้งแต่ Fact-check** ไม่ใช่ข้ามมา QA เลย
